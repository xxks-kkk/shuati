77. Combinations¶

This is a classic backtracking problem. Here is a general thinking to handle this type of problem.

Theorem 8 (Key idea)

We treat backtracking type of problem as a CSP problem. In specific, we solve the problem using \(k\)-nested loops [1] and then reduce the \(k\)-nested loops into recursion format.

Take this problem as an example. If we know \(k\) beforehand (i.e., \(k\) is a fixed constant), we can easily write the following algorithm to solve the problem. Suppose \(k = 2\), then,

            \begin{algorithm}
\caption{$k$-nested loops algorithm}
\begin{algorithmic}
\PROCEDURE{combine}{$n, k=2$}
\STATE $res = \emptyset$ \COMMENT{$res$ contains a list of combination}
\FOR{$i = n$ \TO $1$}
\FOR{$j = i-1$ \TO $1$}
\STATE add $(i,j)$ to $res$ \COMMENT{$k=2$. Thus, each combination contains two elements}
\ENDFOR
\ENDFOR
\RETURN $res$
\ENDPROCEDURE
\end{algorithmic}
\end{algorithm}

Because we know \(k = 2\), we only have two nested loops in our algorithm. However, in reality, we don’t know \(k\) beforehand (i.e., \(k\) is a variable fed into our algorithm), we don’t know how many nested loops we need to write explicitly in our algorithm. This naturally leads to recursion [2].

We can translate Algorithm 2 into recursion as follows (again, for illustration purpose, we assume \(k = 2\)):

              \begin{algorithm}
  \caption{$k$-nested loops algorithm (version 2)}
  \begin{algorithmic}
  \PROCEDURE{combine}{$n, k=2$}
  \STATE $tmp = []$
  \STATE $res = \emptyset$
  \FOR{$i = n$ \TO $1$}
    \STATE $tmp \mathrel{+}= i$ \COMMENT{append $i$ to $tmp$. $tmp = [i]$}
    \FOR{$j = i-1$ \TO $1$}
      \STATE $tmp \mathrel{+}= j$ \COMMENT{$tmp = [i,j]$}
      \STATE add $tmp$ to $res$
      \STATE $tmp \mathrel{-}= j$
    \ENDFOR
    \STATE $tmp \mathrel{-}= i$
  \ENDFOR
  \ENDPROCEDURE
  \end{algorithmic}
  \end{algorithm}

Algorithm 3 is essentially the same as Algorithm 2: instead of constructing each individual answer at the inner most loop directly, e.g., \((i,j)\), Algorithm 3 uses a local variable \(tmp\) to construct the answer incrementally as the algorithm executs from the outermost loop to the innermost loop.

The key observation on Algorithm 3 is that if we look at each individual loop, they are identical to each other. After the loop condition, the algorithm first adds \(i\) to \(tmp\) (i.e., \(tmp \pluseq [i]\)), then the algorithm execution goes to inner loop. After inner loop finishes execution, the algorithm decrements \(i\) from \(tmp\). When the algorithm reaches inner loop, the same pattern repeats, i.e., add element to \(tmp\), let inner loop’s inner loop performs computation, and remove element from \(tmp\). The only special case is when there is no more inner loop to go. In other words, if we number each loop as \(k`th loop, we treat Line 8 of :numref:`k-nested-loops2\) as the \(k = 0\) loop.

With description above, we clearly see the induction step and the base step, which are required to form a recursion. Then, we can transform Algorithm 3 to its recursion form. Along the way, we don’t need to prefix \(k\) any more.

            \begin{algorithm}
\caption{$k$-nested loops algorithm (recursion version)}
\begin{algorithmic}
\PROCEDURE{combine}{$p=n, q=k, res=\emptyset, tmp=[]$}
\IF{q = 0} \COMMENT{We're at the $k=0$th loop}
  \STATE add $tmp$ to $res$
\ENDIF
\FOR{$i = p$ \TO $1$}
  \STATE $tmp \mathrel{+}= i$
  \STATE \CALL{combine}{$i-1, q-1, res, tmp$}
  \STATE $tmp \mathrel{-}= i$
\ENDFOR
\ENDPROCEDURE
\end{algorithmic}
\end{algorithm}

Just for the completeness, this problem can be viewed as a CSP, where there are \(k\) variables and the constraints are no duplicate value appears in the solution to CSP, i.e., the complete assignment to all \(k\) variables. The constraints can be enforced by modifying domain values given the previous variables’ assignments. In TT-style, the algorithm can be written

            \begin{algorithm}
\caption{CSP-style algorithm}
\begin{algorithmic}
\PROCEDURE{combine}{$n, k$}
\STATE impose an ordering on the variables: $X_1, \dots, X_k$
\STATE initialize iterators $I_1, \dots, I_n$ for scanning the domain of each variable
\STATE $i \leftarrow 1$
\STATE $res = \emptyset$
\WHILE{\TRUE}
  \STATE $v =$ \CALL{next}{$I_i$} \COMMENT{The constraint is enforced here by reducing domain of $X_i$}
  \IF{$v = nil$}
    \IF{$i = 1$}
      \STATE \RETURN $res$
    \ELSE
      \STATE $i = i-1$
    \ENDIF
  \ELIF {$v$ instantiate $X_i$} \COMMENT{there is no constraint because the constraint is implemented by domain}
    \IF {$i = k$}
      \STATE add $tmp$ to $res$
    \ELSE
      \STATE $i \mathrel{+}= 1$
      \STATE \CALL{reset}{$I_i$}
    \ENDIF
  \ENDIF
\ENDWHILE
\ENDPROCEDURE
\end{algorithmic}
\end{algorithm}

Certainly, Algorithm 5 is much more verbose than Algorithm 4 and thus, it’s not vise to use it during the interview.
